# Design Decisions (设计决策)

本文档记录项目的核心技术选型决策和风险评估。

## 为什么选择 Apple Vision OCR 而不是 Claude Vision

本项目的核心操作是 **"看到文字 → 点击该文字"**。有两个候选方案：

### 方案对比

| 维度 | Apple Vision OCR (本方案) | Claude Vision (截图 → AI 解读) |
|------|--------------------------|--------------------------------------|
| **坐标精度** | **数学精确**：OCR 返回边界框，确定性公式转换，误差 < 5px | **不可靠**：LLM 视觉定位成功率仅 ~43.9%，偏差 50-100px |
| **延迟** | **~100ms**（本地运行，无 I/O） | **2-5秒+**（需上传截图、等待推理） |
| **成本** | **$0**（macOS 内置） | 每次 ~2000 tokens，30分钟学习 ≈ 72万 tokens |
| **轮询频率** | 每 1-5 秒，无成本压力 | 高频轮询不可行（成本 + rate limit） |
| **确定性** | 相同输入 → 相同输出 | LLM 输出有随机性 |
| **视觉理解** | 仅文字 | 能理解图标、布局、视觉上下文 |

### 为什么 OCR 方案更适合

1. **点击需要精确坐标** — OCR 的天然输出，Claude Vision 的致命弱点
2. **交警12123 的 UI 几乎全是文字按钮** — "去学习"、"确定"、"下一题"全是文字
3. **轮询式自动化对延迟和成本敏感** — 人脸识别弹框需要 1 秒级响应
4. **混合架构取两者之长**：
   - 导航/点击/状态检测：OCR（精确、快速、免费）
   - 考试答题分析：Claude Code 自身（LLM 语义理解交通法规）

## 架构：OCR 精确定位 + 关键字驱动状态机

**核心思路：所有操作基于确定性计算，不依赖 AI 猜测坐标。**

```
截屏 → Apple Vision OCR 提取文字+坐标
  → 关键字匹配判断当前页面状态
  → 取目标文字边界框中心点
  → 坐标转换（归一化→像素→÷Retina→+窗口偏移）
  → cliclick 精确点击
  → 等待 UI 响应 → 循环
```

考试答题时：OCR 提取题目纯文字 → Claude 分析答案 → 用 OCR 坐标点击选项。

## 风险评估

### 法律合规风险

> **重要提醒**：学法减分制度要求驾驶人**本人**参加学习和考试。使用自动化工具辅助学习，特别是自动答题部分，可能违反相关规定。请确保：
> - 人脸识别确实是本人在场
> - 了解并接受可能的法律风险
> - 自动化仅作为辅助工具，学习内容建议本人关注

### 技术风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| App UI 更新导致关键字变化 | 中 | 高 | 关键字集中在 `state_keywords.py`，易于更新 |
| OCR 对特殊字体/颜色识别不准 | 低 | 中 | Apple Vision 中文准确率 ~95%+，使用 fast mode |
| 人脸识别需要用户手动解锁 iPhone | 中 | 高 | `face_monitor.py` 1秒轮询，立即通知用户 |
| 网络中断导致流程中断 | 低 | 中 | 状态机自动重试 |

### 成本

- OCR：**$0**（Apple Vision 本地运行）
- 答题分析：**$0**（Claude Code 自身分析，不调用额外 API）
- 单次运行总成本：**$0**
